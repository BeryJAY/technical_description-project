Introduction
  • Purpose

    • The purpose of this software design documentation is to describe the architecture, system design and the implementation of Biometric based car to driver matching security system for urban plots as described in the requirements document.

    Biometric based car to driver matching security system for urban plots is designed to enhance car security in parking areas.

  • Scope

    • This document describes the architecture, system design and implementation of the driver to car matching security system. The system matchesthe finger print details to the car number plate so that notification is sent in cases of theft. This will help in enhancing car security.

  • Overview

    • In the introduction of the documentation, we describe the purpose of the Biometric based car to driver matching security system for urban plots as described in the system requirements specification.

    The documentation provides the scope of this project explaining its goals, objectives and benefits in brief. The documentation also contains the reference materials which were used.

    In addition, the document contains the system overview which gives the general description of the functionality, context and design of this project.

    The architecture of the system that this project is going to come up with is also explained here. In the Architecture we find modular program structures showing the different relationships of the different modules in the system and how they work together to achieve the complete functionality of the system.

    There is a decomposition description providing the different subsystems in the architectural design.

    The documentation further contains the data design showing the different data modules, component design to look at what each component does in a more systematic way.

    Lastly interface design describing the functionality of the system from the user’s perspective and also the Requirements matrix to provide a cross reference that traces components and data structures to the requirements in the SRS document.

    This system has a mobile application and a desktop application. The mobile application is used by the people who want to use the parking areas to park their cars and these are the drivers of the cars. The desktop application is used by the employees of the parking area to monitor the activities of the parking area.

    The system enables user to see available parking. The activities that a user can perform while using this system include:

  • Logging in
  • Registration
  • Updating user details
  • Viewing available parking areas

    • Functions of the system

    These are the functions of the system;

    1. The system shall be able to capture the biometric information of the car driver and the car number plate.
    2. The system shall be able to match the car’s number plate to its driver’s fingerprint.
    3. The system shall be able to display available parking areas.
    4. Notify the driver who brought in the car at entry about incase another person is taking the car out of the parking.

    Any person with a smart phone who regularly uses parking areas with the system deployed, as well as the parking operators stationed at the parking areas can use this application.

    Architectural Design

    The system follows a client-server architectural style that is divided into three parts, namely;

    1. Client in form of the mobile application that captures the user actions and a desktop application that is installed in the parking area.
    2. Server in form of the Parking Management Server
    3. Database layer, that consists of data repositories.

    The main components of the system that collaborate to achieve the desired functionality include;

    • Client:
      • This is the main user interface in form of mobile application that shows the number of available parking slots. This happens as one is viewing the details of a parking area. Also displays the parking bill they are supposed to pay for a given interval. it also shows the other parking benefits of using the parking area.
    • Parking operator:
      • The person who is able to identify car thefts when the system detects mismatch of car and driver at exit point from the ones that were at car entry, keep watch of those who have not fulfilled payments.
    • Application logic (Parking Management Server):
      • It consists of several modules that are in charge of different responsibilities and these include;
      • Registration module: It is responsible for registering the drivers and parking operators as well as authorizing access to the mobile application through use of login credentials.
      • Matching module: It is responsible for matching the car to the driver’s thumbprint both on entry to and exit from the parking area.
      • Tallying module: This is the module that tallies the fingerprint of the driver and the number plate of the car.
      • Analysis module: This is the module that is responsible for the generation of the weekly reports to the manager.

    • Design Rationale

    The system is based on the Three Tier client/server architectural style that depicts the client as the mobile application and server stub as the Parking Management server.

    Reasons for choosing a three tier Client/Server architecture

  • Heterogeneity: The 3-tier model can work best in a heterogeneous database environment where all the database servers and the interactions with these servers are of different types because the middleware server can be used for provision of data access connections to heterogeneous data sources as a database gateway.
  • High performance: The middleware server provides high speed native access to multiple databases, so performance is improved over database connectivity solutions like Open Database Connectivity. That means one is not embedding SQL (Structured Query Language) programs into client applications and not locking one´s products to specific database vendors.
  • Easily change and maintain each domain without affecting others when the interface, application logic and data service are separated.
  • Implementing a 3-tier application using a database gateway as a middleware server can cause the application logic and business rules achieve database independence. This means that you can use different data sources without modification of the application logic and business rules.

    • Tradeoffs : It increases complexity in building the middleware instead of focusing on the client and server only.

    Data description

    Data mainly from the fingerprint scanner and camera is listed as follows;

    1. Thumbprint details: these details will be matched with those details from the national id registration database and altogether merged into the system database. The details will include name, date of birth and place of birth.

    Human interface design

    Overview of User Interface:

    The user will first register with the system online so that he or she becomes a member. The user interface will be changed from guest to member and it will be confirmed on the screen displayed as “logged in as a member”. This registration is required to get the users details such as phone number since the national id system didn’t capture the number. The system will also be able to scan the driver’s finger print and record the details. On scanning, the user will have to insert his or her thumbprint on the scanner which will have to change color as a feedback that the thumbprint is recorded or not.

    Screen Objects and Actions

  • Data on the scanner is displayed in two colors; red showing that the finger print is false and blue showing that the thumbprint is true.
  • For the web view, the url bar is where the user inserts the link.

    Types of system requirements

    The requirements were divided into functional requirements and non-functional requirements.

    1. Functional requirements
      • FREQ-1: The system shall be able to carry out registration of the drivers who want to use the application.
      • FREQ-2: The system shall be able to monitor the available Parking..
      • FREQ-3: The system shall be able to display Parking space to the drivers..
      • FREQ-4: The system shall be able to match the driver to the car..
      • FREQ-5: The system shall be able to capture the biometric information of the car driver and the car number plate.
    2. Non-functional requirements
      • NREQ-1: The system shall be operative 24hrs every day of the week..
      • NREQ-2: The system should be able to run on all operating systems..
      • NREQ-3: The system shall only be accessed over the internet..
      • NREQ-4: The driver shall receive a notification only when the mismatch of thumbprints has occurred three times..
